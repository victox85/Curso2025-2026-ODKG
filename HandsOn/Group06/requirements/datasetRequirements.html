Dataset Requirements Check — GTFS Static Feed

Selected dataset: GTFS static transit feed for Madrid (Consorcio Regional de Transportes de Madrid - CRTM). This feed consists of multiple text files that are CSV-formatted as per the GTFS specification.

Documentation used:
  • GTFS Specification (Static): gtfs.org/schedule/reference
  • GTFS Validator Rules (MobilityData): gtfs-validator.mobilitydata.org/rules.html
  • CRTM Open License: https://www.crtm.es/licencia-de-uso

Summary

This GTFS feed represents public transport information for Madrid. It is a standard open-data format widely used by transit agencies worldwide. It ships as multiple CSV-like text files (e.g., stops.txt, routes.txt, trips.txt, stop_times.txt, calendar.txt, shapes.txt) that are relationally linked by IDs (e.g., stop_id, route_id, trip_id).

Domain: Smart City — Urban Mobility (Public Transport)
Granularity: Stops, routes, trips, stop-times (schedule-level, with geocoordinates)

Requirement Check

R1. Data are in the smart cities domain Satisfied

Public transport is a core smart-city vertical (urban mobility). The feed describes transit infrastructure (routes, stops, services) used for routing, planning, and civic applications.

R2. Data are available as a CSV file Satisfied

Although files use the .txt extension, the GTFS specification defines them as comma-separated text tables (i.e., CSV). Each table can be opened with spreadsheet tools or parsed with CSV readers.

R3. Data have an open license so it can be published Satisfied

License: Licencia de Datos Abiertos y Estáticos (LDA) del CRTM

Summary: This license allows free and non-exclusive reuse of static public data for both commercial and non-commercial purposes. Redistribution and derivative works (including RDF publication) are explicitly allowed under the following obligations:

  • Must cite CRTM as the source and indicate whether data are raw or transformed.
  • Must preserve metadata such as update date and license conditions.
  • Must not distort or falsify the data's meaning.
  • Cannot imply CRTM endorses the reuse without explicit permission.

Attribution: "Source: Consorcio Regional de Transportes de Madrid (CRTM) - https://www.crtm.es"

R4. Data can be easily linked with generic real-world entities Satisfied

The feed includes stop_lat/stop_lon and human-readable names for stations/stops, which can be aligned to generic entities such as places in OpenStreetMap, Wikidata, DBpedia, or GeoNames. Agencies and routes can also be linked to organization and transport entities. Dates in the calendar files can be linked to standard time ontologies.

R5. Documentation exists for the data (optional) Satisfied

GTFS has extensive public documentation (schema, field constraints, examples) and a well-maintained open-source validator with transparent rules: GTFS Spec, Validator rules. The publisher's portal usually includes a short dataset description and version/date.

R6. Data can come from multiple data sources (optional) Satisfied / Optional

The same GTFS model exists for many cities worldwide. We can potentially combine data from different Spanish cities or merge with complementary smart-city datasets (traffic counts, air quality, weather) using spatial joins. For this assignment we focus on Madrid's CRTM feed; adding more cities later is straightforward.
