Ontology Strategy and Data Mapping

The ontology is designed to create a structured, interconnected representation of the different datasets. It defines the main concepts (as Classes), the attributes of these concepts (as Data Properties), and the relationships between them (as Object Properties).

1. Mapping Datasets to Classes

Each dataset, or a key concept within it, is mapped to a class in the ontology:

Ontology ClassCorresponding Dataset(s)
:SensorLocationRepresents a physical location from Pedestrian Sensor Locations and Microclimate Sensor Locations.
:PedestrianObservationRepresents a single row (a specific count at a specific time) from Pedestrian Monthly Counts.
:MicroClimateObservationRepresents a single row (a set of measurements) from Microclimate Sensors Data.
:PublicHolidayRepresents a holiday entry from the Public Holidays dataset.
:StreetSegmentRepresents the concept of a street, derived from address columns in the location datasets.

2. Mapping CSV Columns to Data Properties

The columns within each CSV file are mapped to data properties that describe the attributes of the corresponding class.

Dataset ColumnOntology PropertyDescription
Installation_Date:installationDateLinks a sensor to its installation date.
Latitude, Longitude:latitude, :longitudeDefine the geographic coordinates of a :SensorLocation.
city, postcode:cityName, :postcodeProvide address details for a :SensorLocation.
Total_of_Directions:pedestrianCountRepresents the total count for a :PedestrianObservation.
Direction_1, Direction_2:direction1Count, :direction2CountStore the individual directional counts.
AirTemperature:airTemperatureStores the temperature for a :MicroClimateObservation.
RelativeHumidity:humidityStores the humidity for a :MicroClimateObservation.
AtmosphericPressure:pressureStores the pressure for a :MicroClimateObservation.
Noise, AverageWindSpeed:noiseLevel, :windSpeedStore other environmental readings.
Public holiday (name):holidayNameThe name of the holiday for a :PublicHoliday instance.
Date (holiday):holidayDateThe date of the holiday.

3. Linking Datasets with Object Properties

Object properties define the relationships between classes, effectively linking the different datasets together into a cohesive knowledge graph.

Object PropertyDomain (Source)Range (Target)Description and Link
:observedAtLocationsosa:Observation:SensorLocationLinks an observation (either :PedestrianObservation or :MicroClimateObservation) to the location where it was recorded. This is achieved by joining via Location_ID or Site_ID.
:locatedOnStreet:SensorLocation:StreetSegmentLinks a sensor's location to a specific street segment, using the street column from the location datasets.
:occursOnHolidaysosa:Observation:PublicHolidayLinks an observation to a public holiday if their dates match. This connects sensor data with the holiday schedule.

Naming Strategy — Urban Sensors

1. Domain and Path

  • Domain: https://linked.melbourne.example/
  • Project Path: lcc/

This creates a common base URI: https://linked.melbourne.example/lcc/. Using a path helps organize different projects and allows for versioning.

2. Namespaces

  • Ontology (for classes and properties, using a hash):
    • URI: https://linked.melbourne.example/lcc/ontology/urban-sensors#
    • Prefix: us:
  • Data/Individuals (using a slash):
    • URI: https://linked.melbourne.example/lcc/resource/

Rule: Use a hash (#) for ontological terms (the vocabulary) and a slash (/) for individuals (the data).

3. URI Patterns

Ontology URIs
  • Ontology Document: https://linked.melbourne.example/lcc/ontology/urban-sensors
  • Classes and Properties:
    • https://linked.melbourne.example/lcc/ontology/urban-sensors#ClassName
    • https://linked.melbourne.example/lcc/ontology/urban-sensors#propertyName
Individual (Data) URIs

The base for all individuals is https://linked.melbourne.example/lcc/resource/.

  • Sensors: /sensor/{sensorId} (e.g., .../resource/sensor/ped-001)
  • Locations (Sites): /location/{siteId} (e.g., .../resource/location/site-1234)
  • Pedestrian Observations: /observation/pedestrian/{sensorId}/{yyyyMMddTHHmm}
  • Microclimate Observations: /observation/microclimate/{deviceId}/{yyyyMMddTHHmm}
  • Administrative Areas: /area/city/{slug}, /area/state/{slug}
  • Streets (Segments): /street/{slug}
  • Holidays: /holiday/{yyyy-MM-dd}-{area}

4. Naming Conventions

  • Classes: Use UpperCamelCase and singular nouns (e.g., Sensor, PedestrianObservation, SensorLocation).
  • Properties: Use lowerCamelCase (e.g., installedAt, hostsSensor, airTemperature, pm25).
  • Individuals: Use stable IDs or descriptive slugs in kebab-case.
  • Units: Specify units in annotations or labels (rdfs:label, rdfs:comment), not in the local name of the property.

5. Recommended Prefixes

@prefix us:      <https://linked.melbourne.example/lcc/ontology/urban-sensors#> .
@prefix sosa:    <http://www.w3.org/ns/sosa/> .
@prefix schema:  <http://schema.org/> .
@prefix dcterms: <http://purl.org/dc/terms/> .
# Suggested base for individuals: https://linked.melbourne.example/lcc/resource/

6. Examples

# Class and Property Definitions
us:Sensor a owl:Class .

us:installedAt a owl:ObjectProperty ;
  rdfs:domain us:Sensor ; 
  rdfs:range us:SensorLocation .

# Individual Data (Instances)
<https://linked.melbourne.example/lcc/resource/location/site-1234>
  a us:SensorLocation ;
  us:streetName "Bourke Street" ;
  us:streetNumber "280" ;
  schema:addressLocality "Melbourne" .

<https://linked.melbourne.example/lcc/resource/sensor/ped-001>
  a us:PedestrianSensor ;
  us:installedAt <https://linked.melbourne.example/lcc/resource/location/site-1234> .

<https://linked.melbourne.example/lcc/resource/observation/pedestrian/ped-001/2025-07-08T0830>
  a us:PedestrianObservation ;
  us:pedestrianCount "1234"^^xsd:integer ;
  us:observedAtLocation <https://linked.melbourne.example/lcc/resource/location/site-1234> .

7. Minimal Diff for Your .ttl File

  1. Change the us: prefix to point to https://linked.melbourne.example/lcc/ontology/urban-sensors#.
  2. Set the ontology's IRI to https://linked.melbourne.example/lcc/ontology/urban-sensors.
