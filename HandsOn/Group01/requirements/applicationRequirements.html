Non-Municipal Facilities in Madrid

Description of the Application

The application’s primary goal is to help people discover, locate, and compare non-municipal facilities (universities, parks and gardens, sports centers and libraries) across Madrid. Users can quickly find what they need by area (district/neighbourhood/municipality), by category (type of facility), or by proximity to a chosen point. For each facility, the app highlights practical details—address, opening hours, contact info—and, when available, nearby public transport (metro, bus, train) to make planning a visit fast and convenient.

Typical tasks include: “show all universities in Chamartín,” “find libraries in Prosperidad,” or “list sports centers near me that are easy to reach by Metro.” The emphasis is on clear, actionable results rather than raw data.

Possible use cases

  • Residents searching for specific facility types within their district or neighbourhood and checking how to get there.
  • Visitors identifying facilities of interest near a location and reviewing nearby metro/bus/train lines.
  • Analysts exploring coverage and diversity of facility types per administrative area to support planning.

Functional Requirements

  • Search and Filter by Area:

    Users can select a district, neighbourhood, or municipality (where applicable) to retrieve all facilities within that boundary. Results must update instantly when the user switches area, and the interface should clearly display the current area context (e.g., breadcrumb or header tags).

  • Filter by Facility Type (Category):

    Users can refine results using a type facet (e.g., university, library, sport center, park). Multiple selections are allowed, and each selection updates the result list and counts. The system should remember selected filters when users navigate between pages or the map.

  • Keyword Search:

    A free-text search matches names and descriptions (e.g., “Universidad Politécnica”, “Parque del Oeste”). It should tolerate minor typos and support simple operators (quoted phrases, AND/OR) to narrow or broaden queries.

  • Map-Based Exploration:

    An interactive map allows panning/zooming to explore facilities visually. The visible viewport determines which results appear in the side list. Clicking a marker opens a concise preview (name, type, address, transport hints) with a link to the full detail.

  • Facility Detail View:

    Each facility page (or drawer) shows: name, type(s), description, opening hours (if provided), address with “open in maps”, contact details (phone, email, website), district/neighbourhood labels, coordinates, accessibility note (if any), and transport hints (e.g., “Metro L7; Bus 27, 53”). Provide consistent formatting and clearly mark missing fields.

  • Public Transport Context:

    When the dataset includes transport information, display it prominently in the detail view. If additional transport datasets are integrated later, show nearby stops/lines within a configurable radius and let users filter results by transport mode (metro/bus/train). Where possible, provide quick links to official timetables or network maps.

  • Proximity and “Near Me” Queries:

    Users can search “near a point” (by typing an address or clicking the map) with a distance slider (e.g., 200–2000 m). Results should be sortable by distance and show an approximate walking time badge when coordinates are available.

  • Administrative Facets and Summaries:

    Present facets and counts by district and neighbourhood (e.g., “Universities (12) in Chamartín”), enabling quick drill-down. Area summary pages should list top facility types and provide a compact overview (e.g., “Most common categories in this area”).

Non-Functional Requirements

  • Performance: The application should load quickly and handle large datasets efficiently, especially when users perform searches or interact with the map.
  • Security: Ensure that data privacy and security measures are implemented when accessing or storing data. Users should not be able to access sensitive or unauthorized information.
  • Accessibility: The application should be fully accessible, following best practices for web accessibility (WCAG standards). This includes support for screen readers, keyboard navigation, and mobile responsiveness.

Application User Interface

The interface should be clean, approachable, and responsive, promoting clarity and speed. The following components will be included:

  • Prominent Search Bar: A forgiving, fast search across names and descriptions with suggestions for common facility types and areas.
  • Intuitive Filters Panel: Clear, multi-select filters for area (district → neighbourhood), facility type, and—when available—transport mode; chips/badges reflect active filters.
  • Two-Pane Layout (Map + Results): An elegant split view with a dynamic map on the left and a sortable, paginated results list on the right; hover/click keeps both in sync.
  • Concise Detail Drawer/Page: A well-structured detail view with readable sections (Overview, Address & Contact, Transport, Hours) and an easy “open in maps” action.
  • Helpful Empty States: Friendly guidance when no results match, suggesting filter tweaks or nearby areas.
