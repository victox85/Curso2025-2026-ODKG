Madrid Air Quality Daily/h2>

Requirement 1: Data are in one of the smart city domains

The dataset contains environmental monitoring time series (air quality) per sampling point within a city. Air quality management is a coresmart-city domain (urban environment).

Requirement 2: Data are available as a CSV file

The selected dataset is available as a CSV file, which is located inside the folder "CSV FILES" uploaded for the first assignment.

Requirement 3: Data have an open license so they can be published

The open license of this data can be found in the following link: License Link

Requirement 4: Data can be easily linked with generic real-world entities

The data can be easily linked with several real-world entities, allowing integration into a Knowledge Graph:

  • Administrative areas: PROVINCIA and MUNICIPIO codes can likely be linked to authoritative identifiers (such as INE codes) and then mapped to external datasets like Wikidata or GeoNames.
  • Stations: ESTACION and MUNICIPIO identifiers may be linked to station metadata in other datasets, such as those from the same open data portal where this dataset comes from.
  • Pollutants: MAGNITUD codes can be mapped to specific pollutants, which are widely represented in Wikidata, such as NO2, PM10, O3, etc.

Requirement 5: Documentation exists for the data (optional)

The source portal provides a codebook/metadata that documents all coded fields:

  • Administrative codes: The PROVINCIA and MUNICIPIO codes correspond to postal codes, which are easy to look up and can be linked to external databases like Wikidata or GeoNames.
  • Stations: A CSV file https://datos.madrid.es/portal/site/egob/menuitem.c05c1f754a33a9fbe4b2e4b284f1a5a0/?vgnextoid=9e42c… Link provides the mapping between ESTACION identifiers and the corresponding station names, enabling easy reference and linkage.
  • Pollutant magnitudes: The MAGNITUD codes are mapped to specific pollutants (e.g., 12 → NO2) and are documented in an external mapping file, which is referenced in the dataset documentation: https://datos.madrid.es/FWProjects/egob/Catalogo/MedioAmbiente/Aire/Ficheros/Interprete_ficheros_%2… Link
  • Measurement techniques: Similar to pollutants, the codes for measurement techniques are documented in the same document as before.
  • Validation flags: The semantics of Vxx are documented, indicating whether a data point is valid (V) or not (N), and any other validation statuses that may be used in the dataset.

The selected dataset is related to smart cities because it defines the different entities and facilities in Madrid that are available on the official website www.madrid.es. This means that all the necessary information, such as location, schedules, or public transport access to each entity, is available online. Therefore, this dataset can be included in the domain of governance and public administration within the smart cities framework, as it involves electronic administration and the management of open data.

Madrid Daily Noise Pollution

Requirement 1: Data are in one of the smart city domains

These data belong to the Smart Environment domain of smart cities, as they result from the technological implementation of connected sensors to monitor pollution, in this case, noise pollution. The public release of this information enables analysis that can lead to policies to improve quality of life. Therefore, it is inderectly related to the Smart Living domain of smart cities.

Requirement 2: Data are available as a CSV file

The selected dataset is available as a CSV file, which is located inside the folder "CSV FILES" uploaded for the first assignment

Requirement 3: Data have an open license so they can be published

The open license of this data can be found in the following link: https://datos.madrid.es/egob/catalogo/aviso-l…

Requirement 4: Data can be easily linked with generic real-world entities

The dataset contains information that can be associated with real-world entities:

  • The NMT column: it is the identifier of the monitoring station. On the same portal https://datos.madrid.es/portal/site/egob/menuitem.c05c1f754a33a9fbe4b2e4b284f1a5a0/?vgnextoid=b8c42… , there is another dataset that provides detailed information about each of these stations (name, location, district, neighborhood, geographic coordinates, and altitude). Therefore, the information from both datasets can be linked. This allows us to obtain complete spatial information that can in turn be linked to external entities (neighborhood, district, street) in Wikidata or DBpedia, or via their geographic coordinates.
  • Temporal information: Year, Month, Day, Period. These attributes can be linked to temporal ontologies, which may enable connections with information such as atmospheric conditions which may be related to the data obtained. For instance, if a specific day is declared a public holiday on the Madrid City Council’s website, an increase in noise could be explained by this factor.
  • Recorded acoustic magnitudes (LAeq, L1, L10, L50...): linkable to sensor ontologies, equivalent vocabularies and standards on noise regulation, or to information about recommended health-related thresholds for noise levels.

Requirement 5: Documentation exists for the data (optional)

The file is accompanied by a PDF document that provides a brief overall description of what is being measured, the time span covered (2014–present), and the measurement frequency. In addition, the document includes a table specifying the data type of each column and an explanation of what each represents. It is worth noting that there are two columns (station and period) whose names in the dataset do not match those in the document; however, it is easy to identify that they refer to the same information, so the dataset’s content becomes fully understandable after reading this additional documentation. Moreover, on the portal that hosts the data, an annual aggregated version is also available, and the information about the stations indicates if any of them are currently affected by a specific issue (e.g., station 3 currently affected by construction work).

Madrid Daily Metereology

Requirement 1: Data are in one of the smart city domains

The dataset belongs mainly to the Smart Environment domain of Smart Cities, with intersections in Smart Living/Health, since it tracks temperature using geolocated stations over time.

Requirement 2: Data are available as a CSV file

The selected dataset is available as a CSV file, which is located inside the folder "CSV FILES" uploaded for the first assignment

Requirement 3: Data have an open license so they can be published

The open license of this data can be found in the following link: https://datos.madrid.es/egob/catalogo/aviso-legal

Requirement 4: Data can be easily linked with generic real-world entities

The dataset can be easily linked with real-world entities such as geographic areas (province, municipality, station) and time (days, months, years). Fields like “PROVINCIA”, “MUNICIPIO” and “ESTACION” directly map to real-world locations, referring to administrative areas and physical meteorological stations. Similarly, “ANO”, “MES”, “D01”, “D02”, etc., represent actual calendar dates, making the dataset directly relatable to external sources such as other meteorological records, climate models, traffic patterns or public health data for the same period. In this dataset, which contains meteorological information for Madrid, several attributes can be directly associated with generic real-world entities:

  • Measurement stations, such as Plaza España, Escuelas Aguirre, or Cuatro Caminos, which correspond to physical monitoring sites across the city.
  • Geographical entities, including province, municipality, and neighborhood, which situate each measurement station in its real location.
  • Meteorological variables, such as temperature, humidity, wind direction, or precipitation, which represent physical environmental conditions that can be observed and compared with global standards.

Requirement 5: Documentation exists for the data (optional)

The dataset is fully documented by the Dirección General de Sostenibilidad y Control Ambiental. It explains the meaning of each field (province, municipality, station, parameter, measurement technique, year, month, day), the coding of meteorological variables (e.g., temperature, humidity, wind speed, precipitation), and the mapping of station codes to real locations. It also specifies the validation rules, indicating that only data marked with the code “V” are valid. This ensures the dataset is standardized, interpretable, and suitable for integration into Smart City applications. This documentation is available at: https://datos.madrid.es/FWProjects/egob/Catalogo/MedioAmbiente/DatosMeteorologicos/Ficheros/Interpretaci%C3%B3n_datos_meteorologicos.pdf
